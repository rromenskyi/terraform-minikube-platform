### Added
- **Optional `mail.dmarc_rua` knob for additional mail domains.** A domain yaml
  with `mail.submission_only: true` can now also set `mail.dmarc_rua: <address>`;
  the emitted `_dmarc` TXT then carries `rua=mailto:<address>` so aggregate
  DMARC reports flow somewhere that actually receives mail. Default (unset)
  keeps the previous report-less record â€” pointing `rua` at a non-existent
  mailbox would just bounce reports. Pairs with publishing an MX through the
  generic per-domain `dns:` record list when an additional domain takes
  inbound mail (relay + Stalwart provisioning stays an operator concern).
- **Vault Phase 1+2 â€” operator-managed secret store via `modules/vault`.** Server StatefulSet (raft single-node, auto-unseal via init Job + postStart hook on a `vault-bootstrap` Secret) was Phase 0 already; this round wires the in/out paths so the cluster actually uses it. New shape:
  - **Phase 1 â€” bootstrap Job + `vault-config-operator` (vco)**. A one-shot `vault-bootstrap` Job runs after init, mounts the bootstrap Secret to read the root token, enables the `kubernetes/` auth method, writes a `vault-config-operator-admin` policy, and binds vco's controller-manager SA to that policy. From there vco reconciles every other Vault-side concern via CRDs â€” operator never touches Vault config by hand again. vco helm release lives in its own `vault-config-operator` namespace; `enableCertManager: true` provisions its webhook + metrics serving certs through the existing platform cert-manager. A long-lived SA token Secret (`vault-token-reviewer`) backs `auth/kubernetes/config.token_reviewer_jwt` so Vault TokenReview calls use the platform's `vault` SA (carries `system:auth-delegator`) regardless of which consumer is logging in.
  - **Phase 2 â€” Zitadel OIDC self-serve + per-tenant policies + VSO**. New `module "vault_oidc"` callsite emits the Zitadel application via the existing `module.zitadel-app` pattern (operator + per-tenant project roles declared via the `roles` input, slug-hyphens normalised to underscores in role keys). vco then renders three CRD families: `JWTOIDCAuthEngineConfig` (against the platform's Zitadel issuer), `Policy operator` (full sudo for the operator) + per-tenant `Policy tenant-<slug>-rw` (RW on `secret/data/tenants/<slug>/*`), and the matching `JWTOIDCAuthEngineRole` CRs that bind each policy to the corresponding `urn:zitadel:iam:org:project:roles` claim value. Tenant list = `distinct(slug)` across `local.projects` â€” every project's slug automatically gets a Vault subtree + policy + OIDC role, dormant until the operator grants the matching Zitadel role to a real user. The `hashicorp/vault-secrets-operator` (VSO) Helm release lands alongside, with a cluster-default `VaultConnection` + `VaultAuth` so per-tenant `VaultStaticSecret` CRs can reference the default by name. VSO impersonates a `vault-secrets-operator-controller-manager` SA in the *consuming* namespace, so the engine emits that SA per project ns when at least one secret in that project is in vault mode. Vault's k8s-auth role (`vso`) accepts the SA across every namespace via a `targetNamespaceSelector` Exists matcher; blast radius stays contained because the bound policy (`vso-tenant-read`) is read-only on `secret/data/tenants/*`.
  - **`storage_class` knob on the vault module.** Default `""` keeps the static hostPath PV (node-local; survives pod restart, not node loss; pinned to whichever node first attached). Set to `"longhorn"` (or any dynamic-provisioning SC) to drop the hostPath PV and let the cluster provision a network-backed volume â€” Vault then survives node loss + reschedules cleanly. Switching is destructive (new PVC binds an empty volume; raft data on the old hostPath is no longer referenced); operator's responsibility to take a `vault operator raft snapshot save` before flipping. Note: shamir keys don't survive a cross-storage restore â€” a fresh init is required after switch, vco re-pushes every CRD-managed config + policy + role automatically afterwards, only operator-uploaded KV data needs re-paste.
- **Engine vault-mode for operator secrets â€” `vault: true` in per-project `secrets:` block.** Third mode for `modules/project`'s `secrets:` entries, alongside the existing random-shared and literal-mode (`var.operator_secret_values[<name>]`) shapes. When yaml has `vault: true`, engine emits a `VaultStaticSecret` CR pointing at the convention path `secret/data/tenants/<project_slug>/<name>` instead of a literal `kubernetes_secret_v1`. VSO syncs whatever's at that Vault path â†’ k8s Secret with the same name in the project namespace; chart consumers' `envFrom` is unchanged. Tenant uploads values via Zitadel SSO with the `tenant_<slug>` role grant â€” operator out of the loop entirely after the initial role grant. Vault path is convention-only (no `.env` round-trip, no `TF_VAR_operator_secret_values` declaration needed for vault mode).
- **Engine vault-mode for argocd repo deploy keys â€” implicit when not in `var.argocd_repo_ssh_keys`.** `argocd_repos.tf` refactor: each `argocd_bootstraps:` entry's `repo_ssh_key_id` is resolved by checking the legacy `var.argocd_repo_ssh_keys` map first (literal fallback), and falls through to vault mode (`secret/data/tenants/<project_slug>/argocd-deploy-keys/<id>`) when not present. Engine emits a `VaultStaticSecret` with `destination.transformation.templates` to combine the operator-supplied `sshPrivateKey` from Vault with the engine-known static `type=git` + `url=<repo_url>` so the rendered Secret matches Argo CD's repository-Secret schema. `excludeRaw = true` drops VSO's default `_raw` JSON dump (otherwise breaks Argo CD's parse). Per-tenant: `<project_slug>` owns its repo subtree â€” same Zitadel `tenant_<slug>` role lets a tenant self-serve.
- **Engine vault-mode for git-sync deploy keys â€” `git_deploy_keys:` per-env yaml block.** Same per-tenant Vault path layout (`secret/data/tenants/<project_slug>/git-deploy-keys/<id>`) for the OTHER ssh-key flow in the engine: git-sync sidecars on `kind: deployment` components. yaml shape: `git_deploy_keys: { <id>: { host: github.com } }` (host optional). Engine emits `VaultStaticSecret` per entry; VSO syncs into a `kubernetes.io/ssh-auth` Secret named `git-deploy-key-<id>` in the project namespace, with templating that combines the operator-supplied `sshPrivateKey` from Vault with the engine-known `known_hosts` line for `<host>`. The legacy flat `var.git_deploy_keys` map (with private keys verbatim in `terraform.tfvars`) is removed; operators declare entries in domain yaml + upload keys to Vault.
- **Optional Cloudflare DNS-01 ACME solver on the platform's Let's Encrypt ClusterIssuers** (via the upstream `terraform-k8s-addons` v2.4.0 release). New operator-facing `services.dns01_cloudflare` block in `config/platform.yaml`: `enabled: true` + `dns_zones: [<zone>]` lights up a `dns01.cloudflare` solver alongside the existing `http01` on both ClusterIssuers, gated by cert-manager's `selector.dnsZones` so HTTP-01 stays the default for hosts outside the listed zones. Required for Certificates on hosts that can't satisfy HTTP-01 (direct LB endpoints with no port-80 listener â€” UDP/raw-TCP services bound to a MetalLB VIP). Reuses the existing `TF_VAR_cloudflare_api_token` value via a new sensitive `var.cloudflare_api_token` declaration (provider still picks the token from the env var directly; this declaration only feeds the engine-emitted `kubernetes_secret_v1.cloudflare_acme_token` in `cert-manager` namespace).
- **Per-tenant Vault path convention.** Every secret-like artefact the engine emits in vault mode lives under `secret/data/tenants/<project_slug>/<kind>/<id>` â€” `<kind>` âˆˆ `argocd-deploy-keys` / `git-deploy-keys` / arbitrary names from the project's `secrets:` block. Slug = the project's `slug` field (= the domain part of `<slug>-<env>`). Vault policies + Zitadel roles match this layout 1:1, so granting a Zitadel user the `tenant_<slug>` project role under the `vault` Zitadel project is the only operator action ever needed to delegate ownership of an entire tenant subtree.
- **Vault-mode for ARC GitHub PATs â€” `vault: true` on `services.github_runners.scale_sets` entries.** Third auth path alongside the legacy `.env`-bound `TF_VAR_github_runner_tokens` map and the externally-managed `github_secret_name` reference. Engine emits a `VaultStaticSecret` per vault-mode scale set pointing at `secret/data/platform/github-runner-tokens/<scale-set-key>` (one expected data key: `github_token`); VSO syncs into `<scale-set-key>-github-pat` Secret in the scale set's namespace. Same Secret name as in `.env`-mode â€” chart consumption is unchanged. Engine also emits the `vault-secrets-operator-controller-manager` ServiceAccount per scale-set namespace that contains at least one vault-mode entry (VSO impersonation prerequisite). Modes are mutually exclusive; plan-time precondition catches double-wired entries. Preferred path going forward â€” operator rotates PAT via Vault UI without TF re-apply, no `.env` exposure.
- **`modules/seafile` â€” file storage with library-based sync, OIDC SSO, behind Traefik.** Single-pod Deployment of upstream `seafileltd/seafile-mc:13.x` (all-in-one bundle: Seahub Django UI + ccnet + fileserver + memcached binary, Redis as canonical cache backend since 13). Backed by the platform's shared MySQL (Seafile 13 CE is MySQL-only, Postgres unsupported upstream) and Redis. One-shot setup Job pre-creates the three Seafile databases (`ccnet_db`, `seafile_db`, `seahub_db`) plus a scoped `seafile` MySQL user via the cluster's MySQL root credential â€” root password stays out of the running pod's bootstrap Secret. OIDC integration via `seahub_settings.py` ConfigMap rendered from a Zitadel client (separate `module "seafile_oidc"` callsite, same pattern as `module.argocd_oidc`). Auto-provisions Seafile users on first SSO login. Traefik IngressRoute splits `/seafhttp/*` (raw blob fileserver, `StripPrefix` middleware) from everything-else (Seahub UI). Bundled Caddy bypassed â€” Traefik does TLS termination at the cluster boundary. Operator drives toggle + tuning via `services.seafile` block in `config/platform.yaml`; bootstrap admin password surfaces via `terraform output -raw seafile_admin_password` for first login (rotation through Seahub UI thereafter).
- **CoreDNS host overrides â€” `services.coredns.host_overrides` map.** New root-level resource emits a `coredns-custom` ConfigMap in `kube-system` carrying a CoreDNS `hosts` block; k3s CoreDNS auto-imports `*.override` files from that ConfigMap so the new entries front the regular `forward` chain with `fallthrough` on unmatched names. Use case: in-cluster Pods resolving a public hostname to a private IP â€” for example, sending outbound mail through a relay whose TLS cert is issued for the public hostname but whose only reachable address from inside the cluster is a private mesh IP. Public DNS records untouched; only Pods inside this cluster see the override. Empty map (default) emits nothing.
- **`modules/security-scan` â€” continuous CVE scanning of platform-system images.** New module wires upstream `trivy-operator` (Aqua Security) Helm release as the scanner layer plus a weekly CronJob as the snapshot layer. Trivy-operator watches Pods cluster-wide, dispatches per-image scans, and emits `VulnerabilityReport` CRDs scoped to a static allowlist of platform-system namespaces (vault, zitadel, postgres, redis, ollama, etc.); tenant project namespaces are out of v0 scope. The CronJob runs weekly (default Sunday 04:00 UTC), collects every active VulnerabilityReport, formats HIGH + CRITICAL findings into `inventory/cve-report.md`, and force-pushes to a stable branch + opens a PR against the platform repo when the report changed since last snapshot (silent no-op otherwise). HostPath PV pinned to a stateful tier node persists trivy's ~700 MB vulnerability DB across operator pod restarts. GitHub PAT (scope `repo`) lives in Vault under `secret/data/platform/github-deploy-tokens/security-scan` â€” vault-mode pattern, no `.env` round-trip. Operator toggles via `services.security_scan.enabled` in `config/platform.yaml`.

### Changed
- **`modules/redis` `kubernetes_deployment_v1.haproxy` â€” checksum/secret + checksum/configmap pod-template annotations + `tcp-check send-lf` for AUTH.** Two latent bugs surfaced when an unrelated apply forced the haproxy Deployment to redeploy: (1) HAProxy 3.x `tcp-check send` does NOT expand `${VAR}` inside its argument, so the literal string `${REDIS_PASSWORD}` was being sent to valkey; valkey replied `WRONGPASS`, every backend marked DOWN, and frontends served `<NOSRV>` (`SC` flag). Fixed by switching to `tcp-check send-lf AUTH %[env(REDIS_PASSWORD)]\r\n` â€” log-format expansion reliably inlines `%[env(VAR)]` regardless of context. (2) `envFrom: secretRef` is read once at container start, ConfigMap volume mounts auto-refresh on disk but haproxy only reads its config at process start â€” without a checksum-driven rollout trigger, any password rotation or chart-config edit would sit unused on running pods until manual restart. Two `checksum/...` pod-template annotations now sha256-key off the Secret data and the ConfigMap data; TF re-rolls haproxy whenever either changes.
- **Engine null-label adoption across nine more modules** (`mysql`, `postgres`, `redis`, `ollama`, `vault`, `zitadel`, `stalwart`, `roundcube`, `longhorn`). Each module now accepts `var.context` (default `null`) and chains its own `module "label"` off the platform-wide context emitted by the root `module.platform_label`. Every k8s resource the modules own gets `labels = merge(local.tags, { existing })` â€” existing-wins on collisions preserves selector-bound `app = "<X>"` keys safely. Pure metadata.labels updates (in-place mutable on every resource type the modules emit), zero destroy / replace.

### Added
- `modules/redis` opt-in **Valkey Sentinel HA topology** via the new `var.sentinel` block (operator-side `services.redis.sentinel.enabled` in `config/platform.yaml`). Default `false` preserves the current single-StatefulSet path with zero behaviour change. When `true`, the module switches to a Bitnami `valkey` Helm release in `architecture: replication + sentinel.enabled` mode (1 primary + N-1 replicas with Sentinel orchestration), plus our own HAProxy Deployment in front. HAProxy uses sentinel-aware health checks (`tcp-check expect string role:master` filters to current primary) so the flat `redis.<ns>.svc:6379` Service routes traffic only to the elected primary â€” consumers keep their existing connection strings, no client-side Sentinel-aware library required. ACL changes (per-tenant `SETUSER` from project module Jobs) replicate primary â†’ replicas via Redis 6+ replication. Tunable knobs: `replica_count` (default 3), `quorum` (default 2), `chart_version`, `image_repo` / `image_tag` (Bitnami legacy mirror friendly), `haproxy_image`, `haproxy_replicas`. **Switching `false â†’ true` is a one-way data wipe** (single-instance PVC destroyed when its for_each collapses; tenant ACL Jobs need re-trigger to repopulate per-tenant users on the fresh chart deploy). Module remains backward-compatible â€” opt-in only.

### Changed
- **BREAKING (data wipe)** `modules/redis` swaps `redis:7-alpine` â†’ `valkey/valkey:9-alpine` and migrates persistent storage from a manual hostPath PV (`platform-redis-data`, 5Gi `standard` SC) to a Longhorn-backed PVC (`storageClassName: longhorn`, dynamic-provisioned by Longhorn CSI, 3 block replicas across nodes). **Migration path:** old hostPath data on disk is NOT migrated (Longhorn provisions empty volume). On apply: Redis StatefulSet pod restarts, Valkey starts with empty data + reseeds the `default` user via the existing `seed-users-acl` initContainer from `REDIS_PASSWORD`. **Per-tenant ACL users (set up by `modules/project` redis-tenant-setup Job) are NOT auto-replayed** â€” operator must `kubectl delete jobs -A -l app=redis-tenant-setup` then re-`./tf apply` to recreate them. ### Fixed
- `modules/redis` and `modules/project` switch Redis from `--requirepass` to `--aclfile /data/users.acl` so per-tenant ACL users survive `redis-0` restarts. Pre-fix every tenant lost its Redis login on the next pod bounce because `ACL SETUSER` only persisted in memory; now the tenant setup Job runs `ACL SAVE` after each `SETUSER`, and a `seed-users-acl` initContainer in the Redis StatefulSet writes the `default` user line on first boot from `$REDIS_PASSWORD` so the platform-root password keeps working too. `requirepass` and `aclfile` are mutually exclusive in Redis (server refuses to start with both); the migration is invisible from a fresh-volume bootstrap and self-heals on first restart for existing volumes (initContainer sees missing/empty `users.acl` and seeds it; tenant Jobs re-apply `SETUSER` + `SAVE`)

### Added
- `modules/component` learns a `sidecars:` map â€” additional containers that run in the same Pod as the component's main container. - `config/components/chat.yaml` now ships `mcp-weather-simple` (https://github.com/rromenskyi/mcp-weather-simple) as a sidecar of Open WebUI on `127.0.0.1:8000` and pre-registers it via `TOOL_SERVER_CONNECTIONS` so the chat model can call `get_current_weather`, `get_forecast`, and `geocode_city` out of the box. The sidecar runs with `MCP_AUTH_TOKEN` intentionally unset â€” loopback inside a single Pod is the trust boundary, so the bearer-token middleware is skipped
- `config/components/chat.yaml` now also ships `open-terminal` (https://github.com/open-webui/open-terminal, `:alpine` variant, ~230 MB) as a second sidecar on `127.0.0.1:8001`, pre-registered via `TERMINAL_SERVER_CONNECTIONS` so the chat model can execute shell commands, manage files and run code through Open WebUI's dedicated Terminal integration. The image's hardened entrypoint (iptables egress firewall + `su-exec` privilege drop) is bypassed via a `command:` override since the Kubernetes NetworkPolicy layer already enforces egress; that lets the sidecar run cleanly as UID 1000 under the Pod's restricted securityContext. The bearer token is shared with Open WebUI's backend through one `env_random: [OPEN_TERMINAL_API_KEY]` entry and reaches `TERMINAL_SERVER_CONNECTIONS` via Kubernetes `$(VAR_NAME)` substitution. Session state lands on an emptyDir at `/home/user` â€” scratch-only, lost on Pod restart, which is the right lifetime for a chat-session terminal
- **Both integrations still need one manual "Verify + Save" click** per tenant in Admin â†’ Settings â†’ External Tools / Integrations on the first fresh data volume until upstream fixes open-webui/open-webui#18140; a `kubernetes_job_v1` that calls the admin API is a straightforward follow-up if the click starts to bite
- `modules/component` new input `image_pull_policy` â€” one of `Always`, `IfNotPresent`, `Never`, or unset to auto-derive from the image reference (the default). Auto-derivation mirrors Kubernetes' own native behavior: a moving tag (`:latest` or an empty/implicit tag) resolves to `Always`, a pinned tag (`:1.2.3`, `:main`, â€¦) or digest pin (`@sha256:...`) resolves to `IfNotPresent`. `modules/project` forwards `try(each.value.image_pull_policy, null)` so a component YAML can set an explicit override under either the top-level `image_pull_policy:` field (main container) or `sidecars.<name>.image_pull_policy` (per sidecar). Unblocks rollouts of moving-tag images (`:latest` stopped refreshing after the first pull under the previous `IfNotPresent` default coming from the kubelet) without forcing every pinned workload to pay the per-Pod-start HEAD

### Changed
- `modules/component` now emits `env_random` values as explicit `env` entries with `valueFrom.secretKeyRef` instead of a single `env_from` block. Functionally equivalent for components that only read these vars â€” but it enables Kubernetes `$(VAR_NAME)` substitution in subsequent `env_static` values, which `envFrom`-sourced vars don't participate in. Consumers that want to reference a random-env key inside a static-env value (e.g. embedding a bearer token in a JSON connection blob) can now do so without Terraform-side glue. The `env_random_keys` input on `modules/component` is populated automatically by `modules/project` from each component's `env_random:` list
- Cloudflare Tunnel secret is now Terraform-generated via `random_password.cloudflare_tunnel_secret` (48 chars, no special) and fed into `cloudflare_zero_trust_tunnel_cloudflared.main.secret` via `base64encode`. One fewer knob in `.env`. **Migration impact on a running platform:** first `terraform apply` after merge sees the new `random_password` value and plans destroy-and-recreate of the tunnel â†’ brief cloudflared reconnect (10-30 s Cloudflare-side blip while the new JWT propagates to the in-cluster `cloudflared` Deployment). DNS `CNAME` records update in place (target changes to the new tunnel UUID without record recreation)

Operators running with `CLOUDFLARE_ZONE_ID` / `TF_VAR_cloudflare_zone_id` in `.env` can safely drop the line. ## [0.2.0] - 2026-04-19

Shared platform services expand beyond MySQL to cover Postgres, Redis, and Ollama, with a root-owned `platform` namespace and per-namespace ResourceQuota driven from YAML. Component model gains generic `env_random` / `env_static` / `security` patterns; routes are decoupled from components.

### Added
- Shared PostgreSQL 16 (`modules/postgres/`) â€” StatefulSet, `postgres-superuser` Secret, per-tenant database + role provisioned via an idempotent psql Job, credentials in each tenant's `postgres-credentials` Secret (`PG_HOST`, `PG_PORT`, `PG_DATABASE`, `PG_USER`, `PG_PASSWORD`, `DATABASE_URL`). Component opts in via `postgres: true`
- Shared Redis 7 (`modules/redis/`) â€” StatefulSet with AOF persistence + ACL enabled, `redis-default` Secret for the platform-root password. Per-tenant ACL user + `~<ns>:*` key prefix gives real cross-tenant isolation; credentials in each tenant's `redis-credentials` Secret (`REDIS_USER`, `REDIS_PASSWORD`, `REDIS_KEY_PREFIX`, `REDIS_HOST`, `REDIS_PORT`). Component opts in via `redis: true`
- Shared Ollama (`modules/ollama/`) â€” StatefulSet + model-pull Job (sha1-named over the models list so list changes rotate the Job), pod-level resources pulled straight from `config/platform.yaml`. Component opts in via `ollama: true`, which injects `OLLAMA_HOST` / `OLLAMA_BASE_URL` / `OLLAMA_API_BASE` through the per-tenant `ollama-endpoint` Secret. No in-cluster auth â€” expose publicly only through a `kind: external` component with BasicAuth
- `config/platform.yaml` toggles for each shared service (`services.{mysql,postgres,redis,ollama}.enabled`). Live file is gitignored per-operator; `.example` is tracked and ships everything off
- `config/components/chat.yaml` â€” Open WebUI component that proxies chat LLM calls and RAG embeddings to the shared Ollama (thin 1 Gi chat pod, heavy models held by the shared inference server)
- `config/components/ollama.yaml` â€” `kind: external` route so the Ollama HTTP endpoint can be exposed through Cloudflare when wanted (pair with `basic_auth: true`)
- Root-owned `platform` namespace (`platform.tf`) â€” shared by every service module, sized via `config/limits/platform.yaml` (12 CPU / 24 Gi on this workstation; Ollama alone can burn 10 CPU during inference)
- Per-namespace ResourceQuota via `config/limits/<ns>.yaml` â€” overrides `default.yaml` for the matching namespace only; keeps tenant budgets in YAML instead of TF code
- Generic component patterns in `modules/component/`:
  - `env_random: [KEY]` â€” Terraform generates a `random_password` per key, stores it in a per-component Secret, env_from'd into the container. Values persist in state across restarts; nothing is committed to YAML
  - `env_static: {KEY: value}` â€” literal env values in the component YAML
  - `security: {run_as_user, fs_group}` + auto `chown-volumes` init container for hostPath mounts (the kubelet doesn't chown hostPath for fsGroup, so the init container does it explicitly)
- `kind: external` components â€” route-only to a pre-existing cluster Service (Grafana in `monitoring`, Traefik dashboard via `api@internal`, shared Ollama in `platform`), no Deployment owned by the project module

- Decoupled route model â€” `routes: {<host_prefix>: <component>}` in each domain YAML. Same component can back multiple hosts, different hosts can back different components; IngressRoute matches every host pointing at a given component with `Host(a) || Host(b) || â€¦`
