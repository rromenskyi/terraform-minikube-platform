# terraform-minikube-platform

Terraform-first multi-domain Kubernetes hosting platform. One domain = one YAML file. Zero-trust public access via Cloudflare Tunnel. Shared platform services (MySQL, PostgreSQL, Redis, Ollama) with per-tenant isolation.

## What you get

- **Multi-domain hosting** on a single cluster (minikube OR k3s)
- **Per-project isolation**: namespace + resource quota + per-tenant DB/Redis ACL user
- **Shared MySQL 8.0** with auto-provisioned DB + user per project (opt-in per component)
- **Shared PostgreSQL 16** with auto-provisioned database + role per project (opt-in per component)
- **Shared Redis 7** with per-namespace ACL user + key prefix (opt-in per component)
- **Shared Ollama** (LLM inference) + pre-pulled models (opt-in per component)
- **Open WebUI** as a reusable component — per-tenant chat UI that proxies the shared Ollama
- **Cloudflare Tunnel** for public HTTPS access (no open ports on the host)
- **TLS** via Let's Encrypt + cert-manager
- **Traefik** as ingress controller
- **Grafana + Prometheus** for monitoring (`kube-prometheus-stack`)
- **HTTP BasicAuth** at Traefik level, per-component, random password generated by terraform
- **Per-component security context** (`run_as_user`, `fs_group`) + auto init container that chowns hostPath mounts
- **Persistent storage** on the host filesystem (survives cluster re-creation)
- **`./tf bootstrap-minikube` / `./tf bootstrap-k3s`** to bring up everything from scratch in one command

## Architecture

```
Internet
  │
  ▼
Cloudflare (DNS + TLS + Tunnel)
  │
  ▼ (tunnel, HTTP)
cloudflared (2 replicas, `ops` namespace)
  │
  ▼ forwards by `ingress.hostname` rule
Tenant Service  (Deployment in `phost-<slug>-<env>`, created by modules/component)
  │           OR
  ▼           cross-namespace Service for `kind: external` components
  —           (Grafana in `monitoring`, Traefik dashboard via `api@internal`, Ollama in `platform`)
```

Every hostname is declared inside `config/domains/<domain>.yaml` under `envs.<env>.routes`. The map key is the host prefix, the value is a component name resolved from `config/components/`. The same component can back multiple hosts (apex + `www`), different hosts can back different components, and `kind: external` components skip the Deployment entirely and route straight at a pre-existing Service.

### Three-layer stack

The platform composes three upstream Terraform modules (all fetched from GitHub at pinned tags):

1. **Cluster** — `terraform-minikube-k8s` (minikube) OR `terraform-k3s-k8s` (native k3s over SSH). Both export the same output signature, so layers 2 and 3 are distribution-agnostic.
2. **Addons** — `terraform-k8s-addons` — Traefik, cert-manager + Let's Encrypt, kube-prometheus-stack, PSS-labeled namespaces.
3. **Tenant workloads** — this repo — shared `platform` namespace (MySQL / PostgreSQL / Redis / Ollama), per-project namespaces with components, Cloudflare tunnel, DNS records.

### Module structure

```
terraform-minikube-platform/
├── main.tf                     # Wires module.k8s + module.addons + module.project
├── platform.tf                 # Root-owned `platform` namespace + its ResourceQuota
├── mysql.tf, postgres.tf,      # Shared services (toggled via config/platform.yaml)
│ redis.tf, ollama.tf
├── cloudflare.tf               # Tunnel + DNS (fully dynamic from project outputs)
├── cloudflared.tf              # cloudflared Deployment in ops namespace
├── variables.tf, locals.tf     # Platform configuration
├── outputs.tf                  # Projects, credentials, cheatsheet
├── _backend.tf                 # State backend (local or remote)
├── _providers.tf, _versions.tf # Provider config + version constraints
├── tf                          # Bootstrap wrapper script
├── config/
│   ├── platform.yaml           # Which shared services to deploy (gitignored)
│   ├── domains/*.yaml          # Domain configs (gitignored)
│   ├── components/*.yaml       # Reusable component definitions (tracked)
│   └── limits/<ns>.yaml        # Per-namespace resource quota — `default.yaml` is
│                               #   the fallback; any other `<name>.yaml` overrides
│                               #   just for that namespace.
└── modules/
    ├── project/                # Namespace + quota + DB/Postgres/Redis/Ollama hookup
    │                           #   + components + IngressRoutes + BasicAuth middleware
    ├── component/              # Deployment + Service + PV/PVC + ConfigMap +
    │                           #   optional chown init container
    ├── mysql/                  # MySQL StatefulSet (+ Secret, PV)
    ├── postgres/               # PostgreSQL StatefulSet (+ Secret, PV)
    ├── redis/                  # Redis StatefulSet (+ Secret, PV) with ACL enabled
    └── ollama/                 # Ollama StatefulSet (+ PV) + model-pull Job
```

### External dependencies

Three sibling modules are fetched directly from GitHub at pinned releases — no sibling checkout required:

- Layer 1 (cluster bootstrap, pick ONE):
  - [`terraform-minikube-k8s`](https://github.com/rromenskyi/terraform-minikube-k8s) — Option A (minikube), pinned to `v3.0.0`
  - [`terraform-k3s-k8s`](https://github.com/rromenskyi/terraform-k3s-k8s) — Option B (k3s, active default), pinned to `v0.3.1`
- Layer 2 (platform addons): [`terraform-k8s-addons`](https://github.com/rromenskyi/terraform-k8s-addons) — pinned in `main.tf`

`terraform init` downloads the selected modules automatically. To upgrade, bump the `?ref=vX.Y.Z` in `main.tf` and re-run `terraform init -upgrade`.

### Alternative cluster distribution

The cluster module is swappable. `main.tf` has an **Option A — minikube** (commented out) and **Option B — k3s** (active default) block. Both modules export the same output signature (`cluster_host`, `client_certificate`, `client_key`, `cluster_ca_certificate`, `kubeconfig_path`, `cluster_name`, `cluster_distribution`), so layers 2 and 3 are distribution-agnostic.

To switch to k3s (native install via SSH):
1. In `main.tf`, comment out the Option A block and uncomment Option B.
2. In `.env`, set `TF_VAR_ssh_user`, `TF_VAR_ssh_host`, `TF_VAR_ssh_private_key_path`.
3. `terraform init -upgrade`.
4. `./tf bootstrap-k3s` — the root `kubernetes` / `helm` providers lazily open `module.k8s.kubeconfig_path`, so a single apply is enough.

## Prerequisites

- **Docker** (minikube driver) or **a host with SSH + passwordless sudo** (k3s bootstrap)
- **Terraform** >= 1.5.0
- **kubectl**
- **jq**
- **Cloudflare account** with at least one domain

## Quick start

### 1. Configure secrets

```bash
cp .env.example .env
```

Edit `.env` and fill in:

```bash
CLOUDFLARE_API_TOKEN=your-token           # API token (not Global API key)
CLOUDFLARE_ACCOUNT_ID=your-account-id
CLOUDFLARE_ZONE_ID=your-primary-zone-id   # Zone ID of the "infra" domain
CLOUDFLARE_TUNNEL_SECRET=any-random-32-char-string
LETSENCRYPT_EMAIL=your-email@example.com
HOST_VOLUME_PATH=/data/vol                # See Persistent storage below
CLUSTER_NAME=platform

# Only for k3s distribution:
SSH_HOST=127.0.0.1
SSH_USER=your-linux-user
SSH_PRIVATE_KEY_PATH=/home/you/.ssh/id_ed25519
```

### 2. Decide which shared services to run

```bash
cp config/platform.yaml.example config/platform.yaml
# Flip `services.mysql.enabled` / `.postgres.enabled` / `.redis.enabled` /
# `.ollama.enabled` to true for the ones you actually want on this machine.
```

### 3. Configure at least one domain

```bash
cp config/domains/example.com.yaml.example config/domains/mydomain.com.yaml
# Edit the file — see [Domain configuration] below.
```

### 4. Bootstrap

```bash
./tf bootstrap-k3s         # native k3s over SSH (Option B, default)
# or
./tf bootstrap-minikube    # minikube (Option A)
```

This destroys any existing state, reinstalls the cluster from zero, purges stale Cloudflare tunnel + DNS, and runs a single `terraform apply` to converge everything.

### 5. Verify

```bash
terraform output cheatsheet
terraform output -json projects | jq .
```

## Configuration

### Platform-service toggles (`config/platform.yaml`)

The shared platform services are opt-in. The repo's `.example` ships everything off; copy it and flip the bits you want.

```yaml
services:
  mysql:
    enabled: true
  postgres:
    enabled: true
  redis:
    enabled: true
  ollama:
    enabled: true
    models:
      - qwen2.5:7b          # chat LLM
      - all-minilm:latest   # embeddings — used by chat component RAG
    memory_request: 4Gi
    memory_limit:   16Gi
    cpu_request:    200m
    cpu_limit:      "10"
```

Each module (`modules/mysql`, `modules/postgres`, `modules/redis`, `modules/ollama`) collapses to zero resources when its `enabled` flag is off. Outputs resolve to null; tenants that ask for a disabled service fail a precondition with a clear error.

### Domain configuration (`config/domains/*.yaml`)

Each domain gets its own YAML file. Live domain configs are gitignored (they contain zone IDs personal to your setup); the `.yaml.example` template is tracked.

```yaml
name: example.com                             # Domain name (used in hostnames)
slug: example-com                             # URL-safe slug (namespace = {prefix}{slug}-{env})
cloudflare_zone_id: "abc123..."               # Cloudflare Zone ID for DNS records

envs:                                         # Map of env → per-env routing
  prod:
    routes:
      "":      web                            # example.com       → web component
      www:     web                            # www.example.com   → web
      api:     whoami                         # api.example.com   → whoami
      blog:    wordpress                      # blog.example.com  → wordpress (own DB, own storage)
      traefik: traefik                        # traefik.example.com → Traefik dashboard (kind: external, basic_auth)
      grafana: grafana                        # grafana.example.com → shared Grafana (kind: external)
      chat:    chat                           # chat.example.com  → Open WebUI → shared Ollama

  dev:                                        # Second env, different namespace
    routes:
      whoami.dev: whoami                      # whoami.dev.example.com → whoami (env in key, explicit)

limits:                                       # (Optional) override default resource quota per project
  cpu: "4"
  memory: "8Gi"
```

**Hostname generation:** the host prefix from the route map key is used literally. Empty string = apex. No env is injected into the hostname — if two envs of the same domain need distinct URLs, spell out the subdomain in the key (`api.dev: whoami` → `api.dev.example.com`).

**Decoupled routes + components:** a component is deployed iff at least one route targets it. The same component can back multiple hosts (bare + `www`); different hosts can back different components (api → `whoami2`, bare → `whoami`).

### Component configuration (`config/components/*.yaml`)

Reusable component definitions. Tracked in git.

Two kinds are supported:

**`kind: deployment`** (default) — this project owns the workload; the component module creates a Deployment + Service:

```yaml
image: wordpress:6.7-php8.3-apache
port: 80
replicas: 1
health_path: null                   # null = disable HTTP probes (WP setup returns 302)

db: true                            # Provision MySQL database + user
postgres: true                      # Provision PostgreSQL database + role
redis: true                         # Provision Redis ACL user + key prefix
ollama: true                        # Inject OLLAMA_HOST / OLLAMA_BASE_URL

# Map component env var names → keys in the `db-credentials` Secret
env:
  WORDPRESS_DB_HOST: DB_HOST
  WORDPRESS_DB_USER: DB_USER
  WORDPRESS_DB_PASSWORD: DB_PASS

# Literal env vars (static values)
env_static:
  RAG_EMBEDDING_MODEL: all-minilm:latest

# Random env values generated by terraform and kept in a per-project Secret
env_random:
  - WEBUI_SECRET_KEY

# Persistent volumes (hostPath on the node, predictable path)
storage:
  - mount: /var/www/html/wp-content
    size: 10Gi

# Pod securityContext + auto chown init container for the hostPath mounts
security:
  run_as_user: 33
  fs_group:    33

# Apache IP restriction via config_files (mounted ConfigMap)
config_files:
  /etc/apache2/conf-enabled/custom.conf: |
    # Apache config content

# Resource requests/limits (defaults shown)
resources:
  requests: { cpu: "50m",  memory: "64Mi" }
  limits:   { cpu: "200m", memory: "256Mi" }

# HTTP BasicAuth at the Traefik IngressRoute level — random password per
# project exposed via `terraform output basic_auth`.
basic_auth: true
```

**`kind: external`** — a Service already exists elsewhere in the cluster; the component emits only an IngressRoute:

```yaml
# config/components/grafana.yaml
kind: external
service:
  name:      kube-prometheus-stack-grafana
  namespace: monitoring
  port:      80

# Optional BasicAuth gate at the ingress level
basic_auth: true

# Optional IR-side service override for Traefik-internal services like api@internal
# ingress_service:
#   kind: TraefikService
#   name: api@internal
# entry_points: [web]
```

### Resource quotas (`config/limits/*.yaml`)

Every namespace this platform creates looks up `config/limits/<namespace>.yaml` first, and falls back to `config/limits/default.yaml`.

```yaml
# config/limits/default.yaml — applied to every tenant namespace
cpu: "2"
memory: "4Gi"
```

```yaml
# config/limits/platform.yaml — Ollama alone burns 10 CPU during inference
cpu:    "10"
memory: "16Gi"
pods:   20
```

Per-project override via the domain YAML's top-level `limits:` block still works (higher precedence than `default.yaml` for tenant namespaces).

## Cloudflare setup

### Where to find your credentials

1. **Account ID** — [Cloudflare Dashboard](https://dash.cloudflare.com) → any domain → right sidebar → "Account ID"
2. **Zone ID** — Dashboard → select domain → right sidebar → "Zone ID". Each domain has its own.
3. **API Token** — Dashboard → My Profile → API Tokens → Create Token. Permissions:
   - Zone > Zone > Read
   - Zone > DNS > Edit
   - Account > Cloudflare Tunnel > Edit
   - Account > Zero Trust > Edit
   Zone Resources: Include > All zones (or specific ones). Copy the token immediately (shown only once).
4. **Tunnel Secret** — any random string, 32+ chars: `openssl rand -hex 32`

### Add a domain to Cloudflare

1. Dashboard → Add a Site → Free plan
2. Cloudflare shows nameservers — change your registrar's NS records to point there
3. Wait for propagation
4. Copy the new **Zone ID**
5. Create `config/domains/yourdomain.yaml`:
   ```yaml
   name: yourdomain.com
   slug: yourdomain-com
   cloudflare_zone_id: "paste-zone-id"
   envs:
     prod:
       routes:
         "":  web
         www: web
   ```
6. `./tf apply`

## Persistent storage

One knob — `host_volume_path` — is used **verbatim** as the parent of every `hostPath` PersistentVolume. The correct value depends on how the kubelet sees the filesystem:

| Distribution | `host_volume_path` | Why this value |
|---|---|---|
| **Native k3s on Linux** (Option B, default) | `/data/vol` (or any host dir) | k3s runs directly on the host; the kubelet sees the host FS 1:1. |
| **minikube on Linux, `--driver=none`** | `/data/vol` | Identical semantics to k3s. |
| **minikube on macOS, Docker driver** (Option A) | `/minikube-host/Shared/vol` | The minikube VM auto-mounts `/Users` from the Mac host as `/minikube-host`. Put your data at `/Users/Shared/vol` on the Mac; the kubelet sees it as `/minikube-host/Shared/vol`. |
| **minikube on Linux, `--driver=docker`** | a path you bind in explicitly | `minikube mount <host-dir>:<node-dir>` first, then set `host_volume_path=<node-dir>`. |

Data layout under the prefix (same structure on every distribution):

```
{host_volume_path}/
├── platform/mysql/                          # MySQL data
├── platform/postgres/                       # PostgreSQL data
├── platform/redis/                          # Redis AOF
├── platform/ollama/                         # Ollama model cache
├── phost-example-com-prod/wordpress/
│   └── var-www-html-wp-content/             # WordPress uploads, plugins, themes
└── …
```

**On `./tf bootstrap-*`, the cluster and Terraform state are reset but host volumes are NOT deleted.** Re-bootstrapping without wiping the data dirs leaves the databases with the old superuser password while terraform generates a new one — clear the data manually if that's a concern:

```bash
sudo rm -rf /data/vol/platform/{mysql,postgres}/*
```

## Remote state

Local `terraform.tfstate` is the default. To move to Backblaze B2 (S3-compatible):

1. Create a B2 bucket (private) and an application key
2. Edit `_backend.tf`:
   ```hcl
   terraform {
     backend "s3" {
       bucket                      = "myplatform-tfstate"
       key                         = "platform/terraform.tfstate"
       region                      = "us-east-005"
       endpoint                    = "https://s3.us-east-005.backblazeb2.com"
       skip_credentials_validation = true
       skip_metadata_api_check     = true
       skip_region_validation      = true
       skip_requesting_account_id  = true
       skip_s3_checksum            = true
       use_path_style              = true
     }
   }
   ```
3. Add to `.env`:
   ```bash
   AWS_ACCESS_KEY_ID=your-b2-key-id
   AWS_SECRET_ACCESS_KEY=your-b2-application-key
   ```
4. `./tf init -migrate-state`

## The `./tf` wrapper

`tf` is a thin shell wrapper that:
1. Loads `.env` variables as `TF_VAR_*` exports
2. Provides `bootstrap-minikube` / `bootstrap-k3s` / `cloudflare-purge` subcommands

```bash
./tf plan
./tf apply
./tf apply -auto-approve
./tf output -json projects | jq
./tf bootstrap-minikube   # full reset + single-phase apply (minikube)
./tf bootstrap-k3s        # full reset + single-phase apply (k3s)
./tf cloudflare-purge     # delete the tunnel + its DNS records only
```

## Common operations

### Add a component to a domain

1. Add a `routes:` entry under the target env in `config/domains/<domain>.yaml`
2. `./tf apply`

### Add WordPress to a domain

```yaml
# config/domains/mydomain.com.yaml
envs:
  prod:
    routes:
      "":  wordpress        # mydomain.com      → wordpress
      www: wordpress        # www.mydomain.com  → wordpress
```

WordPress gets: its own MySQL database, persistent storage for wp-content, an IngressRoute, env mapping to the `db-credentials` Secret, chown init container so `www-data` (UID 33) can write.

### Expose the Traefik dashboard and Grafana

Route any existing domain at the `traefik` and `grafana` external components:

```yaml
routes:
  traefik: traefik
  grafana: grafana
```

Both components have `basic_auth: true`, so random credentials are generated. Retrieve:

```bash
terraform output -json basic_auth | jq
```

### Deploy Open WebUI against the shared Ollama

```yaml
routes:
  chat: chat
```

The `chat` component is `kind: deployment` (Open WebUI) with `ollama: true` — its env gets `OLLAMA_HOST` + `OLLAMA_BASE_URL` pointing at the shared Ollama Service, and `env_static` pins the embedding model to `all-minilm:latest`. No embedding model runs inside the chat pod.

### View credentials

```bash
terraform output -json mysql      | jq -r '.root_password'
terraform output -json postgres   | jq -r '.superuser_password'
terraform output -json redis      | jq -r '.default_password'
terraform output -json grafana_credentials | jq -r '.password'

# Traefik dashboard login
terraform output -json basic_auth | jq -r '[.[].traefik?.password] | map(select(.)) | .[0]'

# Per-project MySQL credentials
kubectl get secret db-credentials       -n <namespace> -o json | jq '.data | map_values(@base64d)'
# Per-project PostgreSQL credentials
kubectl get secret postgres-credentials -n <namespace> -o json | jq '.data | map_values(@base64d)'
# Per-project Redis credentials
kubectl get secret redis-credentials    -n <namespace> -o json | jq '.data | map_values(@base64d)'
```

### Full re-bootstrap

```bash
./tf bootstrap-k3s     # or bootstrap-minikube
```

Deletes the cluster, resets Terraform state, purges Cloudflare tunnel + DNS records, rebuilds everything. Host volumes preserved.

## Variables reference

| Variable | Default | Description |
|---|---|---|
| `cluster_name` | `platform` | Cluster identifier (used in labels, kubeconfig filename, Cloudflare tunnel name) |
| `host_volume_path` | `/data/vol` | Parent path for hostPath PVs, used verbatim by the kubelet. |
| `namespace_prefix` | `phost-` | Prefix for every tenant-project namespace. Infra namespaces (cert-manager, ingress-controller, monitoring, ops, platform) keep fixed short names. |
| `letsencrypt_email` | *(required)* | Email for Let's Encrypt certificates |
| `cloudflare_api_token` | *(required)* | Cloudflare API token |
| `cloudflare_account_id` | *(required)* | Cloudflare Account ID |
| `cloudflare_tunnel_secret` | *(required)* | Random 32+ char secret for tunnel creation |
| `cloudflare_zone_id` | *(required)* | Primary zone used by the tunnel |
| `ssh_host` / `ssh_user` / `ssh_port` / `ssh_private_key_path` | *(required for k3s)* | SSH target for the k3s installer |

## Known limitations

- **Single-node only** — not for production
- **Re-bootstrap password mismatch** — host volumes outlive state; clear MySQL / PostgreSQL data dirs manually after `./tf bootstrap-*` if you care about the new superuser password working
- **hostPath storage is distribution-coupled** — `host_volume_path` must match how the kubelet sees the filesystem
