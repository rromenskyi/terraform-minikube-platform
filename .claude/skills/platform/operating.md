## `./tf` wrapper

Loads `.env` and either delegates to `terraform` or runs an orchestration subcommand.

```bash
# Pass-through — straight to terraform
./tf plan
./tf apply
./tf destroy
./tf init
./tf <anything>           # state, console, etc — all pass through

# Orchestration subcommands
./tf cloudflare-purge     # delete platform tunnel + orphaned *.cfargotunnel.com CNAMEs
./tf bootstrap-minikube   # phased reset of Option A (minikube)
./tf bootstrap-k3s        # full reset of Option B (k3s) — single-phase
```

`bootstrap-*` stops on an interactive confirmation listing what gets destroyed and what's preserved. `-y` / `--yes` skips for CI. Host volume data under `$HOST_VOLUME_PATH` is NEVER deleted by bootstrap. Cloudflare-side: bootstrap fails preflight if a tunnel with the same name already exists — operator decides purge vs `terraform import`.

The wrapper double-wraps `TF_VAR_` keys correctly (a key already prefixed `TF_VAR_` is NOT prefixed again — this used to silently turn `TF_VAR_ssh_host` into `TF_VAR_tf_var_ssh_host`).

```
# Cloudflare

# CLOUDFLARE_ZONE_ID — REMOVED in unreleased; per-domain in config/domains/*.yaml
# CLOUDFLARE_TUNNEL_SECRET — REMOVED in unreleased; auto-generated by random_password

# Let's Encrypt (must be a real mailbox; RFC-2606 reserved domains rejected)
LETSENCRYPT_EMAIL=roman220220@gmail.com

# Storage prefix — native k3s on Linux, host path == in-node path

```

API token vs API key: the platform's Cloudflare permissions are scoped on a custom token with exactly three permissions — `Cloudflare Tunnel: Edit`, `DNS: Edit`, `Zone: Read`. README has the step-by-step.

## Daily workflows

### Add a new tenant domain

1. Buy domain, add to Cloudflare account, point nameservers, get the per-domain Zone ID.
2. `cp config/domains/example.com.yaml.example config/domains/<your.domain>.yaml`
3. Edit: set `name`, `slug`, `cloudflare_zone_id`, declare `envs.<env>.routes`, optional `envs.<env>.limits`.
4. `./tf plan` → review.
5. `./tf apply`.

The new namespace `phost-<slug>-<env>` appears, with the configured components, IngressRoutes and DNS CNAMEs.

### Add a route to an existing tenant

Edit that domain's `routes:` map. Apply.

### Add a new component definition

`config/components/<name>.yaml` with at minimum: `image`, `port`, optional `env_static`, `env_random`, `storage`, `security`, `sidecars`, `basic_auth`, `db`/`postgres`/`redis`/`ollama` opt-ins.

Reference it from a domain's `routes:` map. Apply. The component is created in any tenant ns whose routes use it.

### Toggle a shared service

Edit `config/platform.yaml` (gitignored — operator-local). E.g. flip `services.postgres.enabled` to `false`. Apply. Module collapses to null; tenants that opted into Postgres fail a precondition with a clear message.

### Add models to Ollama

Edit `services.ollama.models` in `config/platform.yaml`. Apply. The model-pull Job's name is sha1 over the list, so list change rotates the Job; existing models are no-op pulls (idempotent).

### Get tenant credentials

```bash
# DB credentials for a specific tenant
kubectl -n phost-ipsupport-us-prod get secret db-credentials -o yaml

# Redis ACL credentials
kubectl -n phost-ipsupport-us-prod get secret redis-credentials -o yaml

# BasicAuth plaintext for any component
terraform output basic_auth | jq .
```

### Bootstrap from scratch

```bash
./tf bootstrap-k3s
# Confirms blast radius, then:
#  - destroys current TF state (tunnel, k3s)
#  - force-uninstalls k3s over SSH
#  - wipes local terraform.tfstate / .terraform.lock.hcl
#  - purges Cloudflare tunnel + its DNS records (scoped by tunnel UUID)
#  - single-phase terraform apply
```

`./tf bootstrap-minikube` does the same shape against minikube; phased apply because minikube's CNI bootstrap order is fragile.

## Ollama — current state on Roman's box

**Live config (as of session 2026-04-25):**
- Image: `docker.io/ollama/ollama:0.21.1` — **upstream stock**, not the Intel SYCL fork.
- Backend: **Vulkan** — `OLLAMA_VULKAN=1`, `GGML_VK_VISIBLE_DEVICES=1`, `MESA_VK_DEVICE_SELECT=8086:e212` (Arc Pro B50 vendor:device).
- `privileged: true`, `/dev/dri` hostPath mount, `supplementalGroups: [44, 990]` (video, render).
- Generation throughput: ~6 tokens/sec on Arc B50 (256 MB BAR ceiling — see troubleshooting.md).

**Why Vulkan and not SYCL:**
- Intel Arc Pro B50 needs Resizable BAR for SYCL/Level Zero to access full VRAM. The motherboard's UEFI has rBAR locked off.
- Vulkan path doesn't need rBAR — works through the standard 256 MB PCIe BAR window.
- Tradeoff: Vulkan caps at ~6 t/s vs SYCL would do 30+ t/s, but SYCL is simply unreachable until rBAR is enabled.
- Memory has a `project_arc_b50_ollama_sycl.md` entry tracking this; the rBARUEFI mod is documented as **brick risk — not attempting**.

**TF code state:**
- `modules/ollama/main.tf` HEAD is the upstream stock build.
- Uncommitted local diff on the `fix/ollama-context-length-8192` branch is adding **Intel SYCL backend** support (image fork, init container, env vars). That's experimental work-in-progress; live cluster is NOT running it. Live Vulkan config landed via direct `kubectl apply` (violation of "no click-ops" — to be backported into TF when the SYCL experiment concludes).

**Models currently pulled** (per `config/platform.yaml`):
- `qwen3.5:4b` — small, quick experiments (~3 GB)
- `qwen3.5:9b` — chat default, 93.2% tool-calling on mcp-weather eval (~6.6 GB)
- `qwen3:14b` — bigger sibling; better long-horizon (~9 GB)
- `gemma4:e2b` — multimodal (text/image/audio), effective 2B (~7.2 GB)
- `gemma4:e4b` — multimodal, effective 4B (~9.6 GB)
- `all-minilm:latest` — embeddings; shared by every chat pod via `RAG_EMBEDDING_ENGINE=ollama`

**Note on the Intel IPEX-LLM fork:** ships Ollama v0.9.3, can't parse `qwen35` / `gemma4` manifests (HTTP 412 on pull). The pull-Job recognises 412 and logs `SKIP <model>` instead of failing — safe to keep all models in the list across backends.

## Open WebUI (chat component)

Each tenant gets an Open WebUI pod that proxies the shared Ollama:
- Image: `ghcr.io/open-webui/open-webui:v0.9.2` (pinned; was `:main`)
- 1 Gi memory (talks to Ollama, doesn't load models locally)
- Two sidecars in the same Pod:
  - `mcp-weather-simple` on `127.0.0.1:8000` — Open-Meteo backed weather/forecast/geocoding tools (registered via `TOOL_SERVER_CONNECTIONS`)
  - `open-terminal:alpine` on `127.0.0.1:8001` — shell + file management for the chat (registered via `TERMINAL_SERVER_CONNECTIONS`)
- RAG embeddings via shared Ollama (`RAG_OLLAMA_BASE_URL=http://ollama.platform.svc...`, `RAG_EMBEDDING_MODEL=all-minilm:latest`).

**Model filter** hides `all-minilm` (embeddings-only — picking it in chat returns "model does not support chat"). RAG backend still uses it.

**One manual step on first fresh data volume**: Admin → Settings → External Tools / Integrations → click "Verify + Save" once per tool. Upstream bug `open-webui/open-webui#18140`. A `kubernetes_job_v1` calling the admin API is a planned auto-fix when this starts to bite.

```bash
# Cloudflare tunnel pods
kubectl -n ops get pods -l app=cloudflared

# Live ConfigSet for a tenant (when applicable)

# Watch the model-pull Job
```
